# 传统 VPS 部署完整链路

## 综合论点

**传统裸金属/VPS 部署的隐性成本等于「最小可跑七跳链路」与「生产就绪七层能力栈」之间的差集**——DNS、备案、WAF、进程守护、数据备份、CI/CD、监控告警等环节不会被域名和 Nginx 自动附带，且 Docker/K8s/Serverless 仅在有意识选型时才替代其中大部分自建工作。

## 论证

### 两条链路的结构对比

| 维度 | 最小可跑链路 | 生产就绪完整链路 |
|------|-------------|-----------------|
| 入口 | 域名 + HTTPS | 域名 + DNS + [[ICP备案]] + DDoS/WAF（[[Cloudflare]]） |
| 计算 | 单台 VPS | VPS + 安全组 + 系统加固 + 可选多实例 LB |
| 边缘 | [[Nginx]] 反代 | Nginx：SSL 终结 + LB + Gzip/HTTP2 + 缓存头 |
| 应用 | 前端 + 后端进程 | 后端：[[进程守护]] + [[健康检查与探活]] + logrotate |
| 数据 | （常遗漏） | DB + [[Redis]] + 对象存储 + 备份与恢复演练 |
| 发布 | 手动上传 | [[CI-CD]] + 环境隔离 + [[灰度发布与回滚]] + 密钥管理 |
| 持续 | （常遗漏） | 监控告警 + [[Certbot]] 续期 + 漏洞审计 + 操作审计 |

来源：[[传统网站上线链路：从能跑到能稳定跑]]

### 七层遗漏清单（按优先级）

1. **网络与访问层** — 无 DNS/备案/WAF 则国内不可达或易被攻击。
2. **运行时层** — 无守护与探活则崩溃与假死不可自愈。
3. **数据层** — 无 Redis/OSS/备份则 Session、文件与数据恢复成单点。
4. **发布运维层** — 无 CI/CD 与回滚则人为失误放大为全站事故。
5. **安全合规层** — 证书过期、SSH 裸奔、依赖 CVE 为可预期事故。
6. **性能扩展层** — 单点 VPS + 无压缩/缓存则在流量上升时快速触顶。
7. **业务体验层** — 邮件、SEO、错误页与熔断影响用户感知与运维信号。

### 与 PaaS/Serverless 的分野（2026-07 更新）

[[Vercel PaaS 网站上线链路]] 将容器化/Serverless 的抽象判断具体化为 [[Vercel]] 模型：

| 四层 | 传统 VPS | Vercel [[PaaS]] |
|------|---------|----------------|
| 基础设施 | 自建 | ✅ 全托管 |
| 运行时 | [[进程守护]] 自建 | ✅ [[Serverless Functions]] |
| 网络加速 | [[Nginx]]+[[Certbot]]+CDN | ✅ Edge Network |
| 构建发布 | [[CI-CD]] 自建 | ✅ [[Git 原生部署]] |
| 数据层 | 自建 DB/[[Redis]]/OSS | ❌ 第三方（Neon、[[Upstash]] 等） |
| 合规/深度运维 | [[ICP备案]]、logrotate、APM | ❌ 不解决 |

**跨源结论**：PaaS 并未消除 [[生产就绪部署链路]]，而是**将七层中的上四层平台化，下三层（数据、长期运维、合规）外置给第三方或仍回 VPS**。详见 [[传统 VPS 与 Vercel 部署对照]]。
