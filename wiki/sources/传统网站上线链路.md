# 传统网站上线链路：从能跑到能稳定跑

## 核心摘要

- 常见「最小链路」（域名 → HTTPS → VPS → Nginx → 前端 → 后端 → CDN）只满足**能跑起来**，距离**稳定、安全、持续运行**仍缺大量环节。
- 完整链路按七层补齐：网络与访问（DNS、[[ICP备案]]、WAF）、[[进程守护]]、数据层、[[CI-CD]] 与 [[灰度发布与回滚]]、安全合规、性能扩展、业务体验。
- 国内面向公网部署时，[[ICP备案]] 是 80/443 端口的硬门槛，与 DNS、防火墙、[[Cloudflare]] 等访问层配置同等重要。
- [[Nginx]] 在完整链路中承担 SSL 终结、负载均衡、Gzip/HTTP2、静态缓存策略等多重角色，而非单纯反向代理。
- 裸 VPS 部署意味着进程守护、备份、监控、证书续期、密钥管理均需自建；Docker/K8s/Serverless 可托管其中多数环节。

## 关键论点

### 最小链路 vs 完整链路

原文对比了两条路径：

```
最小链路：域名 → HTTPS → VPS → Nginx → 前端 → 后端 → CDN

完整链路：域名 + DNS + 备案 → DDoS/WAF → VPS + 安全组 + 加固
  → Nginx (LB/SSL/Gzip/HTTP2) → 前端 (CDN + 缓存)
  → 后端 (systemd/健康检查/logrotate)
  → 数据库 + Redis + 对象存储 → 备份 + 监控 + 告警
  → CI/CD + 灰度 + 回滚
```

核心判断：**「能跑」与「能稳定跑」之间隔着运维体系的全栈建设**，而非单个组件的替换。

### 网络与访问层：买了域名和 VPS 不等于打通

| 遗漏环节 | 要点 |
|---------|------|
| DNS 解析 | A/AAAA/CNAME 指向 VPS IP；TTL 与解析商稳定性（如 [[Cloudflare]]） |
| [[ICP备案]] | 国内用户必须备案，否则 80/443 被封 |
| 防火墙/安全组 | 显式开放 22/80/443，关闭无关端口 |
| DDoS/WAF | 裸 IP 易被扫；至少 Cloudflare 免费代理或厂商高防 |

### 运行时与进程管理：跑了不等于活着

- 直接 `java -jar` / `python main.py` 在 SSH 断开或崩溃后即挂；需 [[进程守护]]（systemd / supervisor / pm2）与自动重启。
- 环境依赖（JDK、Node、GLIBC）与开发机不一致是「本地能跑、线上不行」的常见根因。
- 缺少 `/health` 与 [[健康检查与探活]] 时，进程假死无法自愈。
- Nginx 与应用日志须 logrotate，否则撑爆磁盘。

### 数据层：前后端之外的数据落点

- 数据库（MySQL/PostgreSQL/Oracle）混部 vs 独立实例：资源争抢与端口暴露风险。
- [[Redis]]：Session、热点、限流等生产标配。
- 对象存储（OSS/S3/MinIO）：用户上传不应存 VPS 本地。
- 全量/增量备份、异地冷备、**恢复演练**——有备份不等于能恢复。

### 发布与运维：代码如何上生产

- 手动 scp 不可持续；需 [[CI-CD]]（GitLab CI / GitHub Actions / Jenkins）。
- 开发/测试/生产环境隔离；密钥不应硬编码（环境变量或 Nacos/Apollo 等配置中心）。
- [[灰度发布与回滚]]：全量覆盖出问题时的秒级回滚预案。

### 安全与合规

- Let's Encrypt 90 天过期，须 certbot + cron/systemd timer 自动续期。
- SSH 加固（禁 root 密码、改端口、fail2ban）、依赖 CVE 审计、auditd/堡垒机操作审计。

### 性能与扩展

- 单台 VPS 单点；流量上升需 SLB/HAProxy/[[Nginx]] Upstream。
- HTTP/2、Gzip/Brotli、静态资源 Cache-Control/ETag、数据库连接池与慢查询监控。

### 业务与体验层

- SMTP/邮件网关（SendGrid、阿里云邮件推送）；SEO（robots.txt、sitemap、SSR）；Nginx 自定义错误页与熔断降级（Hystrix/Sentinel）。

### 容器化与 Serverless 的托管边界

Docker/K8s/Serverless 可托管扩缩容、证书、负载均衡等，但**传统裸金属/VPS 意味着上述全部自建**——这是选型时的隐性成本。
