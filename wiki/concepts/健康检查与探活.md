# 健康检查与探活

## 定义

**健康检查与探活**（Health Check / Liveness Probe）是主动检测应用是否**能正确处理请求**的机制，区别于仅检查进程是否存在。典型实现为 HTTP `GET /health` 返回 200，或 TCP/脚本探针。探活失败时触发 [[进程守护]] 重启、[[Nginx]] upstream 摘流，或 orchestrator 替换实例。

## 为什么重要

原文强调：**进程在但无响应**（假死、死锁、连接池耗尽）时，若无探活，运维只能等用户投诉。这是「跑了后端但没人保证它不死」层级的关键遗漏。健康检查是连接运行时层与发布层（[[灰度发布与回滚]] 放量决策）的传感器。

## 设计要点

- **Liveness vs Readiness**：存活探针决定是否重启；就绪探针决定是否接流量（依赖未就绪时不应进 upstream）。
- **检查深度**：浅检查（进程响应）vs 深检查（连 DB ping）——深检查失败可能导致 cascading 摘除，需权衡。
- **超时与阈值**：连续 N 次失败才动作，避免抖动。
- **Nginx 层**：`max_fails` + `fail_timeout`；或专用 health_check 模块（商业版/open source 替代）。

## 边界与反例

- **/health 永远 200**：未检查关键依赖，假死仍被标记健康。
- **仅应用层无负载均衡层探活**：单机部署时依赖 systemd 重启，但 Nginx 仍可能向启动中的后端转发导致 502。
- **K8s**：readiness/liveness 为平台内置；VPS 需自建等价逻辑。
