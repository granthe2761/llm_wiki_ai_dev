# 灰度发布与回滚

## 定义

**灰度发布**（Canary / Rolling Release）指新版本先对少量流量或部分实例生效，验证无误后再全量推广。**回滚**（Rollback）指发现异常后快速切回上一已知良好版本。二者共同构成 [[生产就绪部署链路]] 发布层的风险控制手段。

## 为什么重要

原文指出：传统部署若「新版本直接全量覆盖」，一旦存在逻辑错误、性能回归或配置失误，故障面等于 100% 用户。至少需有蓝绿或滚动发布的**预案**，理想状态是秒级回滚。

## 常见策略

| 策略 | 机制 | 传统 VPS 实现要点 |
|------|------|------------------|
| **滚动发布** | 逐台替换后端实例 | [[Nginx]] upstream 逐台摘流 → 更新 → 探活 → 加回 |
| **蓝绿部署** | 两套环境，切换入口 | 双目录/双端口，Nginx 改 upstream 指向 |
| **金丝雀** | 按百分比或 Header 分流 | Nginx split_clients 或独立 canary upstream |
| **回滚** | 保留上一制品 + 配置快照 | [[CI-CD]] 制品版本化；数据库 migration 需可逆设计 |

## 边界与反例

- **有回滚包但 schema 不可逆**：代码可秒回，数据库 migration 失败仍可能导致长时间不可用。
- **仅灰度无监控**：小流量未发现的问题在全量时仍会爆发；需配合 [[健康检查与探活]] 与业务指标告警。
- **K8s 原生能力**：Deployment rollingUpdate 内置滚动；裸 VPS 需自行用 Nginx + 脚本实现同等语义。

## 与其他概念的关系

- [[CI-CD]] — 提供可版本化的部署制品，是回滚的前提。
- [[Nginx]] — 流量切换的执行面。
- [[健康检查与探活]] — 灰度期间判断新版本是否可继续放量。

- **[[Vercel]] Instant Rollback** — Preview Deployment 按分支隔离，Production 一键回滚至任意历史部署，语义上简化蓝绿/滚动（见 [[Git 原生部署]]）。
